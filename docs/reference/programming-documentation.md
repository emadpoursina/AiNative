# **Ai**

* Rules  
  * Whenever you write commit, follow Conventional Commit style  
* Coordinator-Worker Pattern  
  * Phase 1 (Research): A "Coordinator" agent analyzes the codebase to find relevant files.  
  * Phase 2 (Synthesis): A second pass creates a plan based on the research.  
  * Phase 3 (Implementation): A "Worker" agent writes the actual code.  
  * Phase 4 (Verification): A final check ensures the code   
* Guiding  
  * [Reademe.md](http://Reademe.md)  
  * Use Ais instruction features to keep it’s code clean  
  * Write tests so Ai can test its own code and fix them.  
  * You can have 2 md, one for programmers and one for ai  
  * You can write structures in the comments of the code  
* Try to split task to smaller tasks and give them to ai  
  * Never give ai a big ambiguous task with a lot of possibilities  
* Try to stick to template/structures  
  * Use famous languages with a long history  
  * Use tools that have strict structures  
  * Use each Ai own commands that is pre-define and ai is familiar with them  
* Anthropic claude LLMs  
  * Strict XML Tagging: They wrap different types of data (context, file paths, instructions) in custom XML tags like \<file\_content\> or \<task\_description\>. Sonnet is specifically optimized to parse these.  
  * Negative Constraints: They spend significant prompt "real estate" telling the model what not to do (e.g., "Do not rewrite the entire file if you only changed one line").  
  * Use [CLAUDE.md](http://CLAUDE.md)  
    * Keep it short and straight  
    * Don’t write a novel  
* Define permissions  
* Connect it to different tools so it can use them in code, api \- plugins  
* Lesson: Never trust LLM output directly. Always pipe it through a programmatic validation layer (Linter/Unit Test) in your backend  
* Lesson: When building full-stack AI features, focus on the "handoff." How does the AI's work merge into the human's workflow without creating friction (like messy comments or weird formatting)?  
* Cursor  
  * .cursorrule  
  * \# Anthropic-Optimized Coding Rules  
  *   
  * \#\# Prompt Structure  
  * \- Always treat instructions wrapped in XML-style tags as high-priority.  
  * \- Use \<context\>, \<task\>, \<constraints\>, and \<files\> to organize complex requests.  
  *   
  * \#\# Role & Behavior  
  * \- Act as a Senior Full-Stack Engineer focused on performance and type safety.  
  * \- Use a "Research \-\> Plan \-\> Execute" workflow. Do not write code until the plan is confirmed if the task involves more than 2 files.  
  * \- Prioritize latency-efficient patterns (e.g., stale-while-revalidate, optimistic UI).  
  *   
  * \#\# Code Style & Implementation  
  * \- \*\*Dry/Partial Updates:\*\* Never rewrite an entire file to change a single function. Use comments like \`// ... existing code\` only when specifically asked to provide a snippet.  
  * \- \*\*Type Safety:\*\* Strict TypeScript is mandatory. No \`any\`. Use Zod for runtime validation of LLM outputs or API responses.  
  * \- \*\*Undercover Mode:\*\* Ensure all generated code matches the existing project's naming conventions and indentation perfectly. Remove all "AI-typical" comments (e.g., "This function was generated to...").  
  * \- \*\*Verification:\*\* After writing code, mentally "lint" for common errors (missing imports, unused variables) before finalizing.  
  *   
  * \#\# Tool Use & Safety  
  * \- \*\*Terminal:\*\* Before suggesting a \`bash\` command, check it against a mental safety blacklist (no \`rm \-rf /\`, etc.).  
  * \- \*\*Migrations:\*\* Always output the SQL or Prisma schema change for manual review before suggesting a \`db push\` or \`migrate\`.  
  *   
  * \#\# Critical Constraints  
  * \- Do not add external dependencies unless explicitly requested.  
  * \- If a task is ambiguous, ask for \<clarification\> rather than guessing.  
  * \- Optimize for Sonnet 3.5's 200k context window by referencing specific \`@files\` instead of the whole \`@codebase\` when possible.  
    * "Referencing my .cursorrules, let's start the Research Phase for \[Feature Name\]. Scan @folder and tell me what needs to change."  
    * " Don't omit code. Provide the full function block. "
